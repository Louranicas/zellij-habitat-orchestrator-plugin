## P0.G9 — cmd_pipe Security Audit

### Finding: The described command-injection vector does NOT exist in v0.1.0.

**Threat model from plan §Phase 4:** *"cmd_pipe accepts arbitrary CLI strings; if any payload reaches `run_command`, that's command injection in a permissioned WASM plugin."*

### Pipe flow (traced)

```
zellij pipe -p habitat-plugin.wasm -n <cmd> -- <payload>
  └─ HabitatDashboard::pipe (main.rs:162)
      └─ HabitatEvent::PipeCommand { name, payload }
          └─ dispatch_event → CmdPipe::handle_event (cmd_pipe.rs:121)
              └─ CmdPipe::dispatch (cmd_pipe.rs:61)   ← payload terminates here
                  └─ PipeLogEntry stored in VecDeque (cmd_pipe.rs:55)
                      └─ rendered via format! in CmdPipe::render (cmd_pipe.rs:130)
```

### Why payload can't reach `run_command`

- `run_command` has exactly two call sites, both in `main.rs`:
  - `main.rs:123` — inside `Event::Timer` via `self.bridge.poll_due(...)`
  - `main.rs:148` — inside `BareKey::Char('r')` via `self.bridge.poll_due(0.0, ...)`
- Both paths consume `BridgeClient::endpoints`, which is populated *only* by `register_sources(m.data_sources())` at plugin init.
- `CmdPipe::data_sources()` returns `Vec::new()` (cmd_pipe.rs:181) — CmdPipe never contributes endpoints.
- `BridgeClient::poll_due` hard-codes the argv: `["curl", "-s", "--max-time", "2", "--connect-timeout", "1", &ep.url]` (bridge-client lib.rs:54, 73) — the URL is the only user-controlled byte string and it comes from `DataSource.url`, set at module construction, not from runtime input.

**Conclusion:** payload is architecturally firewalled from the subprocess boundary.

### Residual vector (MEDIUM): ANSI / control-char injection via render

- `CmdPipe::dispatch` interpolates payload into strings that later reach the terminal via `println!` in `HabitatDashboard::render`:
  - `format!("{}/sphere/{}", self.pv2_url, payload)` (cmd_pipe.rs:86) — payload in URL path stored as `entry.target`
  - `format!("unrecognized command: {command}")` (cmd_pipe.rs:101) — command name in error detail
  - `entry.detail = payload.chars().take(80).collect()` (cmd_pipe.rs:68) — first 80 chars of payload stored verbatim
- `CmdPipe::render` then emits these via `RenderLine::new(format!(" {color}{icon}{R} {B}{:<18}{R} ...", truncate(&entry.command, 18), truncate(&entry.target, 24), truncate(&entry.detail, ...)))` (cmd_pipe.rs:161)
- `truncate` (in `habitat-core/src/render.rs`) limits *visible character count* but does not strip ANSI escape bytes. A payload containing `\x1b[2J\x1b[H` clears the screen; `\x1b]8;;http://…\x07` starts a clickable hyperlink OSC.
- `main.rs:175 format!("ok")` for the cli_pipe_output reply is clippy-bait (`useless_format`) but has no security surface — it only returns to the CLI caller, not the render buffer.

### Severity

| Class | Severity | Rationale |
|---|---|---|
| Command injection via `run_command` | **Not Present** | architecturally firewalled |
| ANSI/control-char injection via render | **Medium** | corrupts display, exposes terminal-emulator parser bugs; any local Zellij user can trigger via `zellij pipe` |
| Log-flood DoS via high-rate pipe | **Low** | `VecDeque` capped at MAX_LOG=50, memory-bounded |
| Unsanitized command name | **Low** | same class as payload, lower bandwidth |

### Recommended fix (defer to Phase 4)

In `CmdPipe::dispatch` (cmd_pipe.rs:61), sanitize `command` and `payload` at entry:

```rust
fn sanitize(s: &str, max: usize) -> String {
    s.chars()
        .filter(|c| !c.is_control() && !matches!(*c, '\u{1b}' | '\u{7f}'))
        .take(max)
        .collect()
}
```

Apply: `let command = sanitize(command, 64); let payload = sanitize(payload, 128);` then operate on the sanitized strings only. Add two unit tests:
- payload with `\x1b[2J` is preserved as literal chars with the escape stripped
- payload length capped at 128 regardless of input size

**Not doing this in P0** — Phase 4 is the right location (module hardening). P0 was the audit; the audit is complete.
